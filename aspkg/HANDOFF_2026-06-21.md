## 2. 数据模型

### 2.1 `source.json` 格式

文件位置：`aspkg/source.json`（编译时嵌入二进制，首次运行写入磁盘）

```json
{
  "version": 1,
  "builtin": {
    "<软件名>": {
      "aliases": ["别名1", "别名2"],
      "category": "工具|开发工具|浏览器|社交|办公|效率增强|网络工具|影音|多媒体|安全|游戏",
      "detect": {
        "display_name": "注册表 DisplayName 匹配",
        "publisher": "可选，提高匹配精度"
      },
      "versions": {
        "版本号": {
          "urls": {
            "installer": ["https://..."],
            "portable": ["https://..."]
          }
        }
      }
    }
  },
  "local": {}
}
```

**detect 字段的作用**：不依赖 as 历史安装记录，直接通过注册表检测软件是否已安装。匹配算法见 `software.rs` 的 `scan_registry_installed()`。

### 2.2 关键结构体（`software.rs`）

```rust
SoftwareDatabase { version, builtin, local }
SoftwareEntry { aliases, category: Option<String>, detect: Option<DetectConfig>, versions }
VersionEntry { urls: HashMap<String, Vec<String>> }  // installer / portable
InstallRecord { version, r#type, install_path, install_time }
RegistryInfo { display_name, version, publisher, install_path, uninstall_string }
DetectConfig { display_name, publisher: Option<String> }
```

## 3. 核心功能实现

### 3.1 `as list` — 列出软件（`cmd/list.rs`）

**两种模式**：
- **默认（网格模式）**：按分类分组，自适应终端宽度多列显示
  - 分类标题用 `bold_cyan`
  - 已安装用 `bold_green`
  - 未安装/未匹配用 `gray`
  - `as list -a` 显示所有 source 条目（已安装绿色、未安装灰色混排）
- **表格模式**（`--type` / `--path`）：列对齐，截断长名称

`list.rs` 和 `software.rs` 核心函数调用链：
```
run()
  → read_all_entries()        读取 source.json
  → read_installed()          读取 installed.json
  → scan_registry_installed() 扫描注册表 + 匹配
  → print_grid_installed()    按分类分组 + grid_print 渲染
```

**来源列显示顺序**：`注册表` 在前，`源` 在后（注册表+源），而不是源+注册表。

### 3.2 `as install` — 安装软件（`cmd/install.rs`）

支持：
- `as install <软件名>` → 查找 source 后下载安装
- `as install <软件名>=<版本>` → 指定版本
- `as install <URL>` → 交互式判断 + 本地扩展写入
- `as install -r list.txt` → 批量安装
- `--portable` / `--installer` → 强制安装类型

安装流程：
1. `find_software()` → 匹配 name/aliases
2. `select_version()` → 选择版本（未指定时自动选最新，用 `cmp_versions` 比较）
3. `select_install_type()` → 选择 installer 或 portable
4. 下载（`net` crate）
5. `installer::install_portable()` 或 `installer::install_installer()`
6. `record_install()` → 写入 installed.json（安装版跳过写入）

**已安装检测交互逻辑**（重要）：
- 检测已安装：同时查 `installed.json`（便携版）和注册表（安装版）
- 版本比较：通过 `registry_version` 将注册表版本映射回 source key，再与最新版本比较
- 已是最新版 → 提示 "当前已是最新版本，是否切换到其他版本？[y/N]"
- 非最新版 → 提示 "是否更新/切换版本？[Y/n]"（默认更新）

**便携版安装流程**（`installer::install_portable`）：
1. 如果目标目录已存在，先 `remove_dir_all` 清理旧文件
2. `detect_archive_type()` → 通过魔数检测文件真实类型（不依赖扩展名）
3. 根据类型解压（zip → `extract_zip`，7z → `extract_7z`，其他 → 直接复制）
4. `find_entry_exe()` → 递归查找主 exe
5. `create_shim()` → 在 `%LOCALAPPDATA%\aminos\bin\` 创建 `.bat` 快捷桩

**`extract_zip` 双重解压**：
- 方法一（主）：`Expand-Archive -Path '...' -DestinationPath '...' -Force`
- 方法二（备）：`.NET ZipFile::ExtractToDirectory('...', '...')`（2 参数版）
- 两种方法都通过 `is_nonempty_dir()` 验证结果

**安装版安装流程**（`installer::install_installer`）：
1. 静默运行安装器
2. 如果静默安装失败，询问用户是否直接运行安装器显示 GUI
3. 安装后通过 `DetectConfig` 查注册表验证是否真实安装成功

### 3.3 `as uninstall` — 卸载软件（`cmd/uninstall.rs`）

支持两种卸载途径：
1. **as 管理的软件** → 删除 installed.json 记录
   - portable：删除目录
   - installer：通过注册表 UninstallString 卸载
2. **注册表检测到的软件**（未用 as 安装但有 detect 匹配）
   - 通过 `detect_from_registry()` + UninstallString 执行卸载
3. **安装版只走注册表**：installed.json 中无安装版记录，卸载时直接查注册表

### 3.4 注册表检测（`software.rs` + `sys/src/registry.rs`）

入口：`scan_registry_installed(entries)`

流程：
1. `sys::registry::scan_all_installed()` → 扫描三个 Uninstall 注册表位置
2. 过滤 SystemComponent=1 和 ParentDisplayName 的条目
3. 对每个注册表条目的 display_name，遍历 source entry 的 detect 字段进行子串匹配
4. 返回 `(matched: HashMap, unmatched: Vec)`

**匹配规则**（`software.rs`）：
```
display_name 忽略大小写含子串匹配 (contains)
publisher: 两边都有时才校验，一方没有则跳过 publisher 条件
```

### 3.5 CLI 帮助（`main.rs`）

自定义彩色中文输出，命令名和别名动态对齐。格式：
```
  as 软件包管理器

  用法: as <命令> [参数]

  命令:
    install   (i)   安装软件
    uninstall       卸载软件
    list      (l)   列出软件
    freeze          导出已安装清单
    update          更新 as 自身
```

## 6. 关键设计决策

### 6.1 为什么是独立 workspace？

`aspkg` 使用独立 workspace（自含 `[workspace]`），与根目录的 `aminos/ls/pp/ss` 等 workspace member 隔离。原因是：
- 旧 aminos 项目有大量遗留依赖（terminal_size, sha2, indicatif 等）
- 新 aspkg 精简依赖（只依赖 color, config, net, sys, clap, serde, anyhow）
- 避免旧项目的版本冲突

### 6.2 为什么 source.json 不放到外部仓库？

当前编译时 `include_str!("../source.json")` 直接嵌入二进制。优点：
- 首次运行立刻可用，无需网络
- 版本与二进制绑定，不存在兼容问题

缺点：更新软件源需要重新编译。后续可考虑 `as source update` 命令从远程拉取。

### 6.3 注册表匹配策略

用 `contains`（子串匹配）而非精确匹配，因为：
- 同个软件不同版本的注册表 DisplayName 可能不同（如 "Python 3.14", "Python 3.13"）
- 发布者名称可能包含版本后缀
- 用户电脑上有很多检测不到 publisher 的条目

### 6.4 颜色方案

```
bold_cyan    → 主标题 / 命令名（醒目）
bold_green   → 已安装软件（操作感）
gray         → 未安装 / 灰色信息
bright_black → 统计数据 / 辅助文字（真正的灰色）
bold_red     → 错误
yellow       → 警告
```

### 6.5 解压策略

- 扩展名不可靠（如 snipaste 的 `win-x64` 无扩展名），使用魔数检测文件真实类型
- 优先使用 PowerShell `Expand-Archive`（兼容性好），回退到 `.NET ZipFile`（2 参数版避免 Encoding 歧义）
- 解压后必须验证目标目录非空，防止静默失败

### 6.6 安装版不写 installed.json

- 安装版软件在注册表中已经有完整的安装信息（版本、路径、UninstallString）
- 无需在 installed.json 中重复维护，避免两个来源不一致的问题
- `record_install()` 对 `"installer"` 类型直接 return
- `read_installed()` 自动过滤并清理历史残留

### 6.7 版本选择策略

- `select_version()` 未指定版本时始终选最新（`max_by + cmp_versions`）
- `cmp_versions` 按数字段逐段比较，准确处理 `1.4.1.1032 > 1.4.1.1026` 等场景
- 避免 HashMap 迭代顺序不可靠导致的"明明有新版却装旧版"问题
